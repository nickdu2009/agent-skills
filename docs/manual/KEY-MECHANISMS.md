# 关键机制
<div class="title-en">Key Mechanisms</div>

## 目标
<div class="title-en">Goal</div>

这一章解释使用者在接入 `agent-skills` 时最容易直接遇到的几个机制，以及这些机制为什么这样设计。

这一章回答的是“这些机制分别是什么、彼此如何分层”。  
它不重复展开方法论层面的取舍，也不替代具体任务里的选型判断；前者看[设计理念](DESIGN-PHILOSOPHY.md)，后者看[决策原因](DECISION-RATIONALE.md)。

## 技能与治理规则的区别
<div class="title-en">Skill vs Governance</div>

这两个词看起来接近，但职责不同。

### 技能模块
<div class="title-en">Skill</div>

`skill` 是一个具体的行为模块，回答的是：

- 在什么情况下启用
- 启用后应该怎样工作
- 输入输出是什么
- 什么情况下应该停用

例如：

- `read-and-locate` 负责在陌生代码区缩小编辑点
- `bugfix-workflow` 负责先诊断再修复
- `targeted-validation` 负责选择最小充分验证

### 治理规则层
<div class="title-en">Governance</div>

`governance` 是项目级规则层，回答的是：

- 哪些技能什么时候该启用
- 它们之间如何交接
- 哪些技能不该同时保持活跃
- 什么情况下要升级、降级或退出

所以，`skill` 更像单个行为模块，`governance` 更像调度和约束这些技能模块的规则系统。

### 使用者会直接感受到什么
<div class="title-en">What Users Will Notice Directly</div>

从使用者角度看，这两层的差别不是“目录里多了几个文件”，而是 Agent 的工作方式会不会更稳定。

如果只有 `skill`，你得到的是一组可被读取的行为说明。  
这意味着 Agent 有机会参考这些约束，但不一定知道：

- 什么时候该优先启用哪个技能
- 多个技能之间应该怎样交接
- 哪些技能不该同时保持活跃
- 什么时候应该停止继续升级流程

如果同时存在 `governance`，你得到的就不只是“能读到技能说明”，还多了一层项目级行为约束。  
这会更容易让 Agent 在真实任务里表现出一致的边界控制、计划纪律和验证习惯。

### 只装技能和装上治理规则的区别
<div class="title-en">Skills Only vs Skills with Governance</div>

可以把两者理解成两种不同层级的接入方式。

只装技能时：

- Agent 主要是“看得到这些技能”
- 是否真的按预期触发和衔接，更依赖 Agent 自己的即时判断
- 更适合想先低成本试用这套技能的人

技能加治理规则时：

- Agent 不只是“看得到”，还会多一层明确的项目级路由约束
- 更容易在多步任务里保持统一工作方式
- 更适合团队协作、长期使用、以及希望行为更稳定的项目

所以，两者的核心区别不是“有没有安装成功”，而是“安装完成后，Agent 的工作纪律是否更可预期”。

### 使用者该怎么理解这两层配合
<div class="title-en">How End Users Should Think About the Two Layers</div>

一个更容易理解的比喻是：

- `skill` 像单个操作手册，告诉 Agent 某一类任务该怎么做
- `governance` 像值班规则，告诉 Agent 什么时候拿哪本手册、什么时候换手册、什么时候停下来

如果只有前者，没有后者，Agent 可能“知道很多做法”，但不一定能在正确时机稳定地选对做法。  
把这两层放在一起，使用者看到的就不是更多术语，而是更稳定的执行习惯。

## 为什么 `AGENTS.md` 是路由层
<div class="title-en">Why `AGENTS.md` Is a Routing Layer</div>

`AGENTS.md` 被设计成路由层，而不是技能正文，是为了避免把同一套说明写两遍。

如果把完整技能说明也塞进 `AGENTS.md`，会很快出现这些问题：

- 同一个技能的规则在两个地方重复
- 技能升级时容易改漏
- 治理文件会变得过大，难以维护
- 路由规则和技能细节混在一起，使用者难以分清职责

因此这个仓库采用的是分层结构：

- `AGENTS.md` 负责说明技能何时进入、何时退出、如何衔接
- 每个 `SKILL.md` 负责说明该技能自身怎样工作

这样可以让治理层保持简洁，也让每个技能的说明有独立、稳定的归属地。

## 为什么安装后会看到 `AGENTS.md` 或 `CLAUDE.md`
<div class="title-en">Why You See `AGENTS.md` or `CLAUDE.md` After Installation</div>

对最终使用者来说，最容易困惑的一点是：我明明安装的是一套技能，为什么项目里最后出现的是 `AGENTS.md` 或 `CLAUDE.md`？

原因是，项目安装不只是把技能放进去，还会把一份适配目标平台的治理规则写进项目。  
这份治理规则不是临时手写出来的，而是从仓库里的治理模板生成出来的。

你可以把它理解成：

- `SKILL.md` 是单个技能自己的说明书
- `AGENTS.md` / `CLAUDE.md` 是项目里真正落地的治理规则文件
- `templates/governance/` 里的模板，是这些治理规则文件的来源

从使用者角度，最重要的不是模板内部怎么拆 section，而是理解它带来的结果：

- Agent 不只是“能看到技能”
- Agent 还会多一层项目级触发、交接和停用约束
- 同一个项目里的行为会更一致、更可预期

所以，安装后看到 `AGENTS.md` 或 `CLAUDE.md`，通常不代表“多出了一份技能手册”，而是代表这个项目已经接入了治理规则层。

如果你只是想使用它，记住这一点就够了：  
模板存在的意义，是把仓库里的治理规则稳定地生成到目标项目，而不是让最终使用者逐段维护模板本身。

## 这些机制如何配合
<div class="title-en">How These Pieces Fit Together</div>

对于普通使用者，可以把这几个机制理解成两层：

1. 技能层（`skill`）
   具体行为模块
2. `AGENTS.md`
   项目级路由与治理规则

把这两层分开之后，项目才不会把“技能内容”和“项目级调度规则”混成一团。

如果你是最终使用者，记住一条最简单的判断就够了：

- 想先低成本试用，可以先从技能层开始
- 想让项目里的行为更稳定、更一致，就应该把治理规则一起带上
