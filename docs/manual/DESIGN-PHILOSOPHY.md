# 设计理念
<div class="title-en">Design Philosophy</div>

## 目标
<div class="title-en">Goal</div>

这套 `agent-skills` 不是为了让 Agent 看起来更忙，而是为了让它在真实仓库里少走弯路、少做错事、少制造高成本改动。

这一章回答的是方法论层面的“为什么”，重点解释这套手册背后的整体偏向。  
如果你想看具体机制是什么，去读[关键机制](KEY-MECHANISMS.md)；如果你想看“为什么这次应该这样选”，去读[决策原因](DECISION-RATIONALE.md)。

它的总体偏向很明确：

- 先把问题缩小，再开始动作
- 先形成清晰意图，再开始编辑
- 先做最小改动，再考虑扩大范围
- 先做最小充分验证，再决定是否扩大验证面
- 只有在拆分成本低于协同收益时，才引入并行

这意味着它有意牺牲一部分“看起来很快”的动作，换取更低的误改率、更低的回滚成本，以及更稳定的结果。

## 为什么先缩小范围再行动
<div class="title-en">Why Scope Before Action</div>

先缩小范围再动手，是因为 Agent 最常见的失败不是“不够努力”，而是太早进入错误区域。

如果任务边界还不清楚就开始搜索、阅读、修改，通常会出现这些问题：

- 工作集快速膨胀，注意力被无关文件稀释
- 本来是局部问题，却被扩大成目录级甚至仓库级改动
- 后续验证成本不断上升，因为改动面已经失控

所以这套方法先要求 Agent 明确“这次到底要解决哪一小块问题”，再进入下一步。

## 为什么先计划再编辑
<div class="title-en">Why Plan Before Editing</div>

先计划再编辑，不是为了增加流程感，而是为了防止冲动修改。

当 Agent 在没有明确计划时直接编辑，最容易发生的是：

- 边改边找方向，最后工作集不断变化
- 隐含假设没有暴露，直到改一半才发现前提不成立
- 进度无法表达，只能说“在处理”“快好了”这种无效状态

短计划的价值在于：

- 先说明目标
- 先说明假设
- 先说明准备改哪些文件
- 先说明准备怎样验证

这样后续每一步都有参照物，偏离时也更容易被发现。

## 为什么强调最小可行改动
<div class="title-en">Why Minimal Viable Change</div>

强调最小可行改动，是因为大多数任务真正需要的不是“顺手优化整片区域”，而是用最小补丁解决当前问题。

最小改动的好处很直接：

- 更容易 review
- 更容易验证
- 更容易定位回归来源
- 更容易回滚
- 更不容易意外改变公共接口和兼容边界

这套技能体系不鼓励“顺便清理一下”“顺便重构一下”，因为这些顺手动作常常才是风险真正扩大的开始。

## 为什么强调最小充分验证
<div class="title-en">Why Minimum Sufficient Validation</div>

优先做最小充分验证，是为了先用最低成本确认这次改动有没有站住脚，而不是默认上来就跑最重的验证。

原因有两个：

- 大验证很慢，会拖慢反馈回路
- 当改动还局部时，先做局部验证往往更能说明问题

这不代表拒绝更大范围验证，而是要求验证也遵循“从最小足够开始”的原则：

1. 先验证最直接受影响的面
2. 如果证据不足，再扩大验证范围
3. 如果改动本身影响多个边界，再升级验证策略

## 为什么并行工作需要前提
<div class="title-en">Why Parallel Work Has Preconditions</div>

并行协作不是默认更快。只有当任务可以拆成低耦合子问题时，并行才值得。

如果任务本身高度耦合，强行并行通常会带来额外成本：

- 子任务边界不清，调查结果互相覆盖
- 多个代理读到的上下文不同，结论难以拼合
- 汇总和冲突裁决成本高于并行节省的时间

所以这套方法强调：先判断任务是否真的适合拆，再决定是否并行，而不是把并行当成默认加速器。

## 实际取舍
<div class="title-en">The Practical Trade-Off</div>

从使用者角度看，这套方法论的核心取舍是：

- 少一点“立即开始改”的速度
- 多一点“先看清楚再改”的纪律

它追求的不是一次对话里看起来最激进的产出，而是在连续多次任务里都能保持稳定、可解释、可复查的工作方式。
