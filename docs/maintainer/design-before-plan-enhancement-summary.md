# design-before-plan 技能增强总结

## 背景

用户质疑"只加 design-before-plan 技能就可以了吗？"促使我们重新审视需求设计阶段的完整性。通过系统性分析典型软件工程需求设计阶段的 12 项关键活动，发现初版 design-before-plan 虽然覆盖了核心设计决策流程，但**遗漏了 3 个生产环境中高频出现的关键缺口**。

## 识别的关键缺口

通过 agent 分析和实际工作场景调研，识别出以下缺口：

### 缺口 1: 隐性需求挖掘（P0 优先级）

**问题表现**：
- Agent 实现功能时忽略安全考虑（输入验证、权限检查）
- 性能需求未明确，导致低效实现（N+1 查询、OOM 风险）
- 缺少日志和监控埋点，影响线上问题排查
- 错误场景未充分考虑（超时、熔断、降级）

**触发频率**：⭐⭐⭐⭐⭐ 极高

**解决方案**：在 Step 1 (Requirements clarification) 中新增五维度隐性需求检查清单：
- **Security**: 认证、授权、输入验证、数据清洗、加密
- **Performance**: 延迟指标（p50/p95/p99）、吞吐量限制、资源约束、查询优化
- **Observability**: 结构化日志、指标/计数器、分布式追踪、错误跟踪、告警阈值
- **Resilience**: 错误处理策略、重试逻辑、熔断器、超时配置、优雅降级
- **Operability**: 部署策略、配置管理、功能开关、回滚计划

### 缺口 2: 数据迁移设计（P0 优先级）

**问题表现**：
- 数据模型变更时缺少迁移脚本
- 未考虑大规模数据迁移的性能影响
- 缺少迁移验证和回滚方案
- 新旧数据格式兼容性问题

**触发频率**：⭐⭐⭐⭐ 高

**解决方案**：在 Step 4 后新增 Step 4.5 (Data migration strategy)：
- **Schema 变更识别**：新字段、类型变更、重命名、删除、索引修改
- **迁移路径设计**：
  - Forward migration（old → new）：迁移脚本、数据转换逻辑
  - Backward migration（new → old）：回滚支持、恢复能力
- **复杂度和风险评估**：
  - 数据量（< 1M 行：内联迁移；> 1M 行：后台任务）
  - 停机容忍度（零停机：双写期；可接受维护窗口：stop-the-world 迁移）
  - 数据丢失风险（破坏性变更 vs. 增量变更）
- **迁移验证**：行数校验、校验和比较、抽样验证
- **性能影响**：锁竞争、复制延迟、存储增长

### 缺口 3: 技术债务评估（P1 优先级）

**问题表现**：
- Agent 在实现新功能时绕过现有技术债，导致债务累积
- 错过清理技术债的好时机
- 新设计与现有技术债产生冲突

**触发频率**：⭐⭐⭐ 中等

**解决方案**：在 Guardrails 中新增轻量级技术债务评估条目（非独立步骤）：
- 仅在债务**直接阻塞设计**或清理能**显著简化设计**（减少 3+ 文件爆炸半径）时触发
- 在设计文档中记录但**不混合清理与功能交付**
- 避免与 `minimal-change-strategy` 的"不做 drive-by 清理"原则冲突
- 如果清理价值显著，将债务清理作为**单独增量**纳入计划

## 不建议补充的领域

以下活动经分析后认为**不需要覆盖**：

1. **用户故事分析** — 触发频率低（agent 工作输入通常已经是技术化需求），scoped-tasking Step 0 已部分覆盖
2. **需求冲突解决** — 应升级给用户而非 agent 自行决策（需要业务优先级判断）
3. **设计模式选型作为独立步骤** — 已被方案比较覆盖，模式应自然涌现而非强制应用

## 最终增强方案

### 修改的部分

1. **Execution Pattern Step 1** — 扩展隐性需求挖掘
2. **Execution Pattern Step 2** — 补充设计模式指导（自然涌现）
3. **Execution Pattern Step 4.5** — 新增数据迁移策略
4. **Guardrails** — 新增三组检查：
   - 隐性需求检查（按变更类型触发）
   - 数据迁移检查（按 schema 变更触发）
   - 技术债务评估（轻量级，上下文依赖）
5. **Output Contract** — 扩展示例，添加 `implicit_requirements` 和 `data_migration` 节
6. **Common Anti-Patterns** — 新增"忽略隐性需求"案例

### 保持不变的部分

- 核心执行流程（7 步）
- 设计方案枚举上限（2-4 个）
- 只读探索原则（禁止原型实现）
- 验收标准必须从需求推导（非实现细节）

## 增强效果评估

### 覆盖度提升

| 前期设计活动 | 增强前 | 增强后 |
|-------------|--------|--------|
| 需求深化 | 较强 | **强** |
| 隐性需求挖掘 | 无覆盖 | **强** |
| 数据迁移设计 | 无覆盖 | **强** |
| 技术债务评估 | 无覆盖 | **轻量级覆盖** |
| 设计方案比较 | 强 | **强**（含设计模式指导） |

### 关键改进点

1. **系统化隐性需求检查** — 从"靠经验想起来"变为"按变更类型强制检查"
2. **数据迁移路径规划** — 从"实施时临时处理"变为"设计阶段明确策略"
3. **技术债务权衡** — 从"要么忽略要么过度清理"变为"上下文依赖的轻量级评估"

## 对示例场景的影响

更新了 `examples/design-before-plan-scenario.md`，新增：
- Implicit requirements 节（安全/性能/可观测性/韧性四维度）
- Anti-pattern "忽略隐性需求"（展示生产事故案例）
- Success indicator "识别隐性需求"

## 验证结果

- ✅ 仓库布局验证通过
- ✅ Claude 本地镜像同步完成（18 个技能）
- ✅ 技能在系统中正确注册
- ✅ 所有修改符合技能库规范

## 结论

通过增强 design-before-plan，技能库现在覆盖了从需求澄清到实施验证的**完整工作流**，并填补了生产环境中最关键的三个缺口：

1. **隐性需求挖掘** — 防止安全漏洞、性能问题、可观测性缺失
2. **数据迁移设计** — 防止数据丢失、停机事故、回滚失败
3. **技术债务评估** — 平衡债务清理与功能交付

**不需要创建额外的技能**。单个增强的 design-before-plan 技能即可满足需求设计阶段的完整需求。
