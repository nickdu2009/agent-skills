# Agent 技能库评价报告

> 评价日期：2026-03-28
> 评价对象：agent-skills 仓库（14 个技能 + AGENTS.md + 测试/工具体系）

---

## 总体评价

这是一套架构清晰、设计成熟的 Agent 行为纪律库。它解决的是 LLM 编码 Agent 最核心的问题：**行为纪律**。它不教 Agent "知道什么"，而是教 Agent "怎么工作"。这个定位是它最强的架构选择。

**评分：8/10** — 在 Agent 行为纪律领域，这是一套设计严谨、架构清晰、可用于生产的技能库。主要扣分在于缺少实效验证数据和一些实操层面的边界细节。

---

## 一、设计哲学 — 优秀

整套系统贯穿一条清晰的主线：

> **先限定范围，再制定计划，做最小的改动，做最窄的验证，控制上下文膨胀。**

这个哲学从 `AGENTS.md` 到每一个 `SKILL.md` 是内部一致的。README 里明确说了：

> 这个仓库有意偏向更小的步骤而非更广泛的自动化。这种偏向以一些速度换取了更低的错误率、更低的审查成本、以及当一条工作路线被证明是错误的时候更容易回退。

这恰恰是当前编码 Agent 最需要的权衡——Agent 最常见的失败模式就是**过度自信地大范围修改**。

---

## 二、技能分层 — 清晰合理

14 个技能组织为三个层次：

| 层级 | 技能 | 职责 |
|------|------|------|
| **执行层**（8 个） | `scoped-tasking`、`plan-before-action`、`minimal-change-strategy`、`targeted-validation`、`context-budget-awareness`、`read-and-locate`、`safe-refactor`、`bugfix-workflow` | 约束单个 Agent 在一个有界任务内如何工作 |
| **编排层**（2 个） | `multi-agent-protocol`、`conflict-resolution` | 约束主 Agent 如何协调并行子 Agent |
| **阶段系统**（3 个 + 工具脚本） | `phase-plan`、`phase-execute`、`phase-contract-tools` | Schema 优先的多波次项目执行，带验证器、渲染器和契约 |

依赖关系干净：

- 执行层技能水平组合，任意搭配都合理。
- 编排层在上方，需要时拉入执行层技能。
- 阶段系统自成子系统，`phase-contract-tools` 是唯一的契约权威。

---

## 三、单项技能质量 — 上乘

每个 `SKILL.md` 结构统一，包含：

- **Purpose** — 为什么需要这个技能
- **When to Use / When Not to Use** — 明确的触发条件和排除条件
- **Core Rules** — 核心约束规则
- **Execution Pattern** — 分步执行模式
- **Input/Output Contract** — 输入输出契约
- **Guardrails** — 防护栏
- **Composition** — 与其他技能的组合方式
- **Example** — 具体例子

这个结构统一性非常重要——它意味着 Agent 无论加载哪个技能，都能用同样的方式消费它。

### 特别好的设计点

1. **`bugfix-workflow`**：要求"证据先行"，不允许在没有确认故障域的情况下就开始修改代码。这直接对抗了 Agent 最常见的"看到代码就想改"冲动。

2. **`context-budget-awareness`**：提出了量化的触发阈值（8 个文件、2 次重复读取、3 个假设、3 次无进展操作），并明确说这些是"起始默认值，可调整"。这比模糊的"注意上下文"有效得多。

3. **`conflict-resolution`**：提出了证据质量的优先级排序（直接代码路径证据 > 可复现行为证据 > 日志相关性 > 结构类比），不让"语气最强"的结论赢。

4. **`multi-agent-protocol`**：并行性记分卡（6 个问题，4+ 才并行）是一个简洁有效的决策框架，Tier 1/Tier 2 的分级也很务实。

5. **`phase-contract-tools`**：作为唯一的契约权威，带有完整的 Python 验证器/渲染器和冒烟测试夹具，确保契约漂移可被检测。

---

## 四、组合设计 — 独到

这套系统最巧妙的设计之一是**每个技能都声明了自己的组合伙伴**。例如：

- `bugfix-workflow` 组合 `scoped-tasking`（限定故障域） + `read-and-locate`（快速定位路径） + `minimal-change-strategy`（最小修复） + `targeted-validation`（验证症状）

这意味着技能之间不是扁平列表，而是有明确的**组合拓扑**。README 中的 Mermaid 图就是这个拓扑的可视化。

推荐的"起始组合"也很实用：

> `scoped-tasking` + `minimal-change-strategy` + `plan-before-action` + `targeted-validation`

这四个技能构成了一个最小但完整的行为纪律基线。

---

## 五、测试体系 — 务实但轻量

测试基础设施包括：

- **静态检查**：`sync-cursor-skills.py --check` 检测镜像漂移
- **CI**：GitHub Actions 检查 `SKILL.md` 位置、frontmatter 格式、OpenSkills 安装冒烟
- **场景测试**：5 个 `examples/` 场景作为行为验收测试
- **评分体系**：`skill-evaluation-rubric.md` 提供 0/1/2 三级评分标准
- **触发测试**：`trigger_test_data.py` 中的测试矩阵覆盖了触发准确性
- **阶段系统冒烟**：`phase-contract-tools` 自带 `run_smoke_checks.py` 和 golden files

这个测试思路是正确的——行为型技能无法用单元测试验证，必须通过场景 + 评分的方式。

## 六、与同类方案的对比

| 维度 | 这套技能库 | 典型的 system prompt 工程 | 框架级 Agent（如 SWE-Agent） |
|------|-----------|------------------------|--------------------------|
| **关注点** | 行为纪律 | 角色扮演/输出格式 | 工具调用流程 |
| **可组合性** | 高，每个技能独立可组合 | 低，通常是单块 prompt | 中，通常绑定特定框架 |
| **可迁移性** | 高，跨平台映射清晰 | 低，绑定特定模型/平台 | 低，绑定特定架构 |
| **可测试性** | 中，有场景 + 评分体系 | 低，通常无结构化测试 | 高，有 benchmark |
| **复杂任务支持** | 强（阶段系统） | 弱 | 中 |

这套系统在"可组合的行为纪律"这个定位上，是设计最成熟的方案之一。

## 七、可改进之处

### 1. 技能触发仍然依赖平台的描述匹配

每个 `SKILL.md` 的 `description` 字段是 Agent 决定是否加载该技能的唯一信号。但不同平台（Cursor、Codex、Claude Code）的技能匹配机制不同，实际触发率可能不稳定。

**建议**：补充一份跨平台触发测试结果的基线报告，量化各平台的触发准确率。

### 2. AGENTS.md 与技能之间的边界模糊地带

`AGENTS.md` 中的 Change Rules、Context Budget、Validation Rules 与 `minimal-change-strategy`、`context-budget-awareness`、`targeted-validation` 有大量重叠。评分体系中虽然定义了"简单任务用 AGENTS.md 规则，复杂任务升级到完整技能"的标准，但这个边界在实践中很难控制。

**建议**：在 `AGENTS.md` 中的重叠部分增加明确的"升级条件"标注，例如"当 X 发生时，加载完整技能"。

### 3. 阶段系统的复杂度跳跃

执行层和编排层技能都是轻量的单文件 `SKILL.md`，而阶段系统（`phase-contract-tools`）一下跳到了 13 个 Python 脚本 + 多个 JSON/YAML schema + 冒烟夹具。这个复杂度跳跃意味着：

- 只有在真正需要多波次大型项目执行时才值得引入。
- 对于大多数单任务/单会话场景，阶段系统是过度设计。

**建议**：在 README 中更明确地标注阶段系统的适用场景阈值（例如"预计超过 N 个 PR 或 M 个模块时才考虑"）。

### 4. 缺少"反模式"文档

每个技能都描述了正确行为，但没有系统性地列出**常见反模式**。评分标准中有 Pass/Fail 信号，但那是评审视角，不是执行引导。

**建议**：在每个技能的 `Guardrails` 部分增加 1-2 个"典型反模式"示例，直接展示 Agent 常犯的错误。

### 5. 没有技能卸载/降级机制

系统有详细的"何时加载"逻辑，但没有明确的"何时卸载或降级"机制。在长会话中，Agent 可能同时背负 5-6 个技能的规则，导致上下文膨胀——这恰恰违反了 `context-budget-awareness` 的精神。

**建议**：在 `AGENTS.md` 或新增一个轻量技能中定义"技能降级"规则，例如"当任务进入纯执行阶段时，可以放下 `scoped-tasking` 和 `read-and-locate`"。

### 6. 缺乏真实项目的实效验证数据

测试体系设计合理，但仓库中没有附带任何实际测试报告或转录分析。没有证据表明这些技能在真实项目中被系统性地验证过。

**建议**：补充 2-3 份匿名化的真实转录评分报告，证明技能在实践中的实际效果。

---

## 八、总结

### 优点

- 设计哲学清晰一致：小步骤、证据先行、最小改动
- 技能结构统一，消费方式一致
- 组合拓扑明确，不是扁平列表
- 编排层（多 Agent 协议）设计务实，有量化决策框架
- 阶段系统的契约权威模式（单一来源 + 验证器）是正确的工程选择
- 测试体系的思路正确（行为场景 + 评分标准 + 触发测试）

### 不足

- 触发稳定性未经跨平台量化验证
- AGENTS.md 与轻量技能的边界需要更明确
- 阶段系统复杂度跳跃大，适用场景阈值不够清晰
- 缺少反模式示例和技能卸载机制
- 缺少真实项目的实效验证数据
