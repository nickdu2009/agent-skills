## Step 2 — Read and Locate: Find the code and confirm duplication

**Strongest clue:** The task names three webhook handlers that each normalize the same incoming payload shape.

**Discovery sequence I would follow:**

1. **Find the handlers.** Search for webhook handler registrations — grep for route decorators, handler function definitions, or framework registration patterns (e.g., `@app.route`, `@router.post`, `def handle_*_webhook`).

2. **Read each handler's normalization block.** For each of the three handlers, read just the function body and identify the normalization logic — the lines between "receive raw payload" and "pass normalized data to business logic."

3. **Diff the normalization blocks.** Compare the three blocks line by line. Mark each as:

**Decision point — behavioral differences:**
- If all three are identical or cosmetically different: proceed with extraction.
- If any behavioral difference exists: document it, stop the refactor for that handler, and split it into a follow-up task. Only extract the subset that is truly equivalent.

## Step 4 — Safe Refactor: Execute one structural step at a time

### Step 4a — Write the shared helper

**What I do:** Create `normalize_webhook_payload(raw_payload: dict) -> dict` in the shared module. The body is copied verbatim from one of the three handlers (the "canonical" copy — pick the one with the cleanest variable names). No logic changes.

**Why verbatim:** Copying the exact logic preserves behavior. Renaming variables or "improving" the code in the same step would mix structural change with behavior change.

**Validation (targeted-validation):**
- Run: `python -m py_compile handlers/utils.py` (or equivalent lint/type check).
- Sufficient because: the helper has no callers yet, so no runtime path is affected. We just need it to parse and type-check.

### Step 4b — Switch handler 1

**What I do:** In `order_webhook.py`, replace the inline normalization block (lines 34-52) with:

```python
from handlers.utils import normalize_webhook_payload

normalized = normalize_webhook_payload(raw_payload)
```

The lines above and below the normalization block stay untouched — the handler signature, response codes, and downstream call with `normalized` are already in place.

**Decision point — variable naming:** If the handler used a different variable name (e.g., `clean_data` instead of `normalized`), I either:
- Keep the old name and assign: `clean_data = normalize_webhook_payload(raw_payload)` — zero downstream changes.
- Or rename — but only if it's a local variable with no further reach.

**Validation (targeted-validation):**
- Run: `pytest tests/test_order_webhook.py -v`
- Boundary: only handler 1's tests. The other two handlers haven't been touched yet.
- If tests pass: proceed. If tests fail: the extraction changed behavior — stop and investigate before touching the next handler.

### Step 4c — Switch handler 2

Same pattern as 4b, applied to `payment_webhook.py`. Validate with `pytest tests/test_payment_webhook.py -v`.

### Step 4d — Switch handler 3

Same pattern, applied to `shipment_webhook.py`. Validate with `pytest tests/test_shipment_webhook.py -v`.

### Step 4e — Remove dead code

**What I do:** Check each handler for any imports or local helper functions that were only used by the old inline normalization. Remove them.

**Validation (targeted-validation):**
- Run: `pytest tests/test_order_webhook.py tests/test_payment_webhook.py tests/test_shipment_webhook.py -v`
- Also run lint across the three handler files to confirm no unused imports remain.
- This is the first time I widen validation to all three files simultaneously, because the dead-code removal could affect any of them.

## Step 4: State Invariants and Execute (safe-refactor)

### Invariants (stated before any edit)

1. **Handler signatures** — function name, parameters, return type unchanged
2. **Response codes** — every HTTP status code returned by each handler remains identical
3. **Normalized output shape** — field names, types, default values produced by normalization are byte-for-byte equivalent
4. **Error messages** — any validation errors raised during normalization stay the same
5. **No new dependencies** — the helper does not pull in modules the handlers didn't already use

### Refactor steps (one structural operation each)

**Step 4a — Extract the helper**

Write `normalizeWebhookPayload` as a pure function that takes the raw payload and returns the normalized structure. The implementation is copied verbatim from one of the three handlers (whichever is the most complete/canonical).

*Validation:* The helper compiles/type-checks. No handlers are changed yet — existing tests still pass unchanged. I'd run the handler tests to confirm no regression from the new file alone.

**Step 4b — Switch Handler A**

Replace Handler A's inline normalization block with a call to `normalizeWebhookPayload`. Handler A's signature, response codes, and downstream calls remain untouched.

*Validation:* Run Handler A's tests. Compare: does every test that passed before still pass? If a test fails, the normalization output differs — stop and investigate before proceeding.

**Step 4c — Switch Handler B**

Same operation for Handler B. Replace inline block, keep everything else stable.

*Validation:* Run Handler B's tests.

**Step 4d — Switch Handler C**

Same operation for Handler C.

*Validation:* Run Handler C's tests.

**Step 4e — Remove dead code**

Delete the now-unused inline normalization blocks if any residual lines remain. This is a separate step so the deletion is independently reviewable.

*Validation:* Run all three handlers' tests one final time.

### Decision point: behavior divergence

If during Step 4b/c/d a test fails because one handler's normalization actually differs (e.g., Handler B coerces a field to uppercase while the others don't), I would:

1. **Stop** — not force-merge the divergent behavior into the shared helper.
2. **Document** the divergence explicitly.
3. **Split** into a follow-up task: "Reconcile Handler B normalization difference before sharing helper."
4. **Leave** the divergent handler on its inline logic for now, and only share the helper among the handlers that truly match.

This matches the non-goal: *"If extraction reveals behavior differences between handlers, split into a follow-up task."*
